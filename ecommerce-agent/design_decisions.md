# Design Decisions: Rebuilding with LangChain & LangGraph

This document outlines the architectural decisions and technical reasoning behind rebuilding our store chatbot using LangChain, LangGraph, and the Google Gemini API.

---

## 1. Overview
This project implements an intelligent customer support chatbot for an online e-commerce store. It leverages Google's `gemini-1.5-flash` model combined with external Python tools to look up real-time orders, search products, and inspect product details. By representing the agent flow as a state graph, the chatbot can autonomously run multiple tools, chain search queries together, and formulate friendly, accurate responses without hardcoded rules.

---

## 2. Why LangChain + LangGraph?
Moving from custom keyword-matching python logic to a framework-based agent provides several benefits:
- **Clean Tool Decoration**: LangChain's `@tool` decorator automatically compiles Python function signatures and docstrings into API schemas that the LLM understands, removing the need to write custom parsing code.
- **Gemini Integration**: Using `langchain-google-genai` enables straightforward binding of tools to the Gemini API (`bind_tools`). This standardized interface manages system instructions and parses tool call requests out-of-the-box.
- **Visual Graph Structure**: LangGraph represents the agent's logic flow visually as a state graph containing nodes (actions) and edges (transitions). This makes the execution path transparent and easy to diagram on a whiteboard.
- **Industry Standards**: Both tools are standard in modern enterprise AI development. Learning how to manage states, construct nodes, and link edges is highly valuable for real-world engineering roles.
- **Less Boilerplate**: Compared to writing raw HTTP requests to call Gemini and handling tool execution loops manually, LangGraph manages the recursion loop cleanly while keeping the underlying pattern visible.

---

## 3. How the LangGraph Loop Works
The agent's lifecycle is defined as a graph loop:
1. **AgentState**: A shared memory structure (`TypedDict`) containing a list of conversation `messages`. It uses `operator.add` so that whenever a node returns new messages, they are appended to the list rather than overwriting existing history.
2. **llm Node**: This is the entry point. The message list is sent to Gemini. Gemini decides whether it has enough information to answer the user directly, or if it needs to call one (or more) of the provided tools to gather data.
3. **should_continue Edge**: A conditional router function. It inspects the last message returned by the `llm` node:
   - If Gemini requested tool calls, the router directs the state to the `tools` node.
   - If Gemini returned a standard text response, the router directs the state to `END`.
4. **tools Node**: A prebuilt `ToolNode` that executes the corresponding Python functions (like `get_order` or `search_products`) using the arguments generated by the LLM. The results are packaged into special `ToolMessage` formats.
5. **Looping**: Once the tools execute, an edge directs the flow back to the `llm` node. Gemini receives the original user message, its own tool request, and the results returned by the tools. It then evaluates the data and decides its next move.

This cycle continues dynamically until the model concludes with a final response.

## 4. Tool Chaining Example
Consider the user prompt: *"Is there a cheaper alternative to the shoes I ordered in ORD-1001?"*

Here is how the LangGraph loop resolves this request:
1. **Step 1 (llm node)**: Gemini receives the prompt. It realizes it needs the order details for `ORD-1001` first, so it requests a tool call to `get_order(order_id="ORD-1001")`.
2. **Step 2 (tools node)**: The tool runs, logging the call and returning:
   `{"order_id": "ORD-1001", "customer_name": "Alice Smith", "product_id": "PROD-101", "status": "Shipped", "estimated_delivery": "2026-06-28"}`.
3. **Step 3 (llm node)**: Gemini inspects the result. It sees the product ID is `PROD-101`. To determine the product category or details, it invokes `get_product(product_id="PROD-101")`.
4. **Step 4 (tools node)**: The tool runs and returns the product details:
   `{"name": "Running Shoes X1", "category": "Footwear", "price": 89.99, ...}`.
5. **Step 5 (llm node)**: Gemini sees the product is "Running Shoes X1" in the "Footwear" category costing $89.99. It now calls `search_products(query="Footwear")` or `search_products(query="shoes")` to find alternatives.
6. **Step 6 (tools node)**: The search runs, returning options: "Classic Leather Boots" ($120.00), "Sport Sandals Pro" ($55.00), and "Trail Running Shoes" ($75.00).
7. **Step 7 (llm node)**: Gemini compares the prices. It notices that the "Sport Sandals Pro" ($55.00) and "Trail Running Shoes" ($75.00) are cheaper than the original $89.99 shoes.
8. **Step 8 (END)**: Gemini compiles a friendly reply listing these options and terminates the loop.

## 5. Error Handling
- **Database Misses**: When a customer requests an invalid order ID (like `ORD-9999`) or an invalid product ID, the Python tools capture this and return a dictionary containing an error explanation (e.g., `{"error": "No order found with ID ORD-9999"}`).
- **Model Reasoning over Errors**: Instead of the program raising an exception and crashing, the error dict is fed back into Gemini as tool outputs. The model reads the failure and formulates a helpful response explaining that the order was not found and asking the user to double-check the ID.
- **Safety**: The system prompt instructs Gemini never to make up order statuses, prices, or product details. This prevents hallucinations when tools return empty results.

---

## 6. What I Would Improve with More Time
- **Stateful Memory (Persistence)**: Equip the graph compilation with a checkpointer (like `MemorySaver` or Postgres checkpointer) to preserve conversation history across separate session calls.
- **Semantic Product Retrieval**: Connect a vector store (like FAISS or ChromaDB) and generate embeddings for product descriptions, allowing the tool to find matches even if the search keyword doesn't appear verbatim in the database.
- **Stream Output**: Use Streamlit's `st.write_stream` and LangGraph's event stream API (`graph.astream_events`) to render responses word-by-word, reducing perceived latency.
- **Order Modification Tools**: Add tools to perform state updates, such as modifying shipping addresses or updating delivery details, protected by safety checks.
